ECC Security — Roles, Users & Authorizations
Q1–14
Q1 What is SU53 and when do you use it? ▼
SU53 shows the last failed authorization check for a user — exactly which authorization object or field value is missing.

But it only captures the most recent failed check. If the user clicked around before reporting the error, SU53 might show the wrong thing. In that case, I do a screen-share and use STAUTHTRACE to trace it live.
💡 Always tell the user: run SU53 immediately after the error — before clicking anything else. Otherwise you get wrong data.
Q2 What is the difference between ST01 and STAUTHTRACE? ▼
Both trace authorization failures, but there is one key difference:

ST01 — Only works on the same application server where the user is logged in. First check AL08 to see which server the user is on. If you're on a different server, you'll get zero records.

STAUTHTRACE — Works across all application servers. Select "System-wide trace" and it covers every instance. No server matching needed.
💡 Always prefer STAUTHTRACE — it's simpler and avoids the application server mismatch problem entirely.
Q3 What are the Return Codes (RC) in authorization traces? ▼
Return codes tell you exactly why an authorization check failed:
RCMeaningWhat to do
0Authorization passed ✓Nothing — access is fine
4Object exists but field values missingAdd the missing field value to the role
8User buffer issueAsk user to log off and log back in
12Authorization object completely missingFind the right role in SUIM and assign it
Q4 What is the difference between Master Role and Derived Role? ▼
Use Master/Derived roles when the access (T-codes) is the same but the organizational values are different — like Company Code, Plant, or Sales Org.

Master role — Contains all the T-codes and authorization objects. Org values are left open (not filled).

Derived role — Inherits everything from the master. You only fill in the org values here (e.g., Plant 1000 for India, Plant 2000 for China).

Check the relationship in table AGR_DEFINE, and org level values in AGR_1252.
💡 Never add T-codes directly to a derived role. Same goes for functional values like document types — always update these in the master role.
Q5 What does SU24 do and why is it important? ▼
SU24 defines the link between a T-code and its authorization objects. When you add a T-code in PFCG (Menu tab), the authorization objects that appear in the Authorization tab come from SU24.

The key setting is Proposal value:
  • Yes — The object is pulled into the role automatically
  • No — The object is not pulled in

When you add a new object to a custom T-code in SU24 and want it in an existing role, go to PFCG → Expert Mode → Option 3: "Read old status and merge with new data."

SU24 changes are saved as a Workbench Request because they affect all clients.
Q6 What is the difference between "Change Auth Data" and "Expert Mode" in PFCG? ▼
Change Authorization Data — For new roles. Pulls all default objects from SU24 fresh.

Expert Mode — For existing roles. Gives you three options:
  • Delete everything and recreate from scratch
  • Keep old values as-is
  • Read old status and merge with new data — Use this when SU24 has been updated (e.g., new object added to a custom T-code) and you want those changes to flow into the role without losing existing values.
Q7 What are the 5 user types in SAP and when do you use each? ▼
  • Dialog — Regular business users. Log in through SAP GUI. Password expires (e.g., every 90 days).
  • Service — Shared/functional IDs like Firefighter IDs. No password expiry. Multiple logins allowed.
  • System — Used for background jobs and RFC connections within the same system. No dialog login allowed.
  • Communication — For external RFC calls between different systems. No dialog login.
  • Reference — Used to extend authorizations to other users. Not for direct login.
Q8 What is the difference between User Group in Logon Data tab vs the User tab in SU01? ▼
Logon Data → User Group — Used for authorization checks. For example, you can restrict a security consultant from managing users in a specific group (like China or Pakistan) by removing that group from the S_USER_GRP object in their role. They'll get an "unauthorized" error when they try.

User tab → User Group — Just an organizational label. No authorization check happens here. It's used for reporting and grouping only.
Q9 What is S_TABU_DIS and S_TABU_NAM? What's the difference? ▼
Both control access to SAP tables, but at different levels:

S_TABU_DIS — Controls access at the authorization group level. Tables are grouped together, and you grant access to the whole group. Check which group a table belongs to using TDDAT.

S_TABU_NAM — Controls access at the individual table name level. More precise — you specify exactly which table (e.g., MARA, KNA1) the user can access.

Both are critical authorization objects. Always get proper business approval before assigning them.
Q10 What is the full transport process for role changes? ▼
1
Make role changes in Development system using PFCG
2
Run risk simulation to check for any SOD conflicts before saving
3
Create a Transport Request using SE09 / SE10 and include the role
4
Release the TR and move it to Quality
5
Ask the requester to test in Quality and get their UAT sign-off
6
Email the Basis team to transport to Production
7
Confirm with user in Production, then close the ticket in ServiceNow
Q11 What do the Green, Yellow, Red status lights mean in PFCG? ▼
  • 🟢 Green — All authorization objects are maintained and profile is generated. Role is ready to use.
  • 🟡 Yellow — At least one authorization object has missing or incomplete values. Regenerate the profile.
  • 🔴 Red — Organizational level values are unmaintained. Role won't work correctly until filled in.
Q12 How do you add a custom T-code to the system and to a role? ▼
1
ABAP team creates and registers the custom T-code (via SE93)
2
They share a template with the required authorization objects for that T-code
3
Go to SU24, add the objects, set Proposal = Yes, save (select Workbench TR)
4
Get role owner approval before making changes to the role
5
In PFCG, add T-code to Menu tab. In Authorization tab → Expert Mode → Option 3 to pull in new objects
6
Generate profile, create TR, follow Dev → QA → Prod process
Q13 What authorization objects do you need to create users in SU01? ▼
  • S_USER_GRP — Controls which user groups you can work with
  • S_USER_AUT — Controls what authorization values you can assign
  • S_USER_PRO — Controls which profiles you can assign
  • S_USER_AGR — Controls which roles you can assign to users
Q14 What does PFCG_TIME_DEPENDENCY do? ▼
It's a background job that automatically removes expired role assignments from users. When a role is assigned with a validity end date, this job cleans up those assignments once the date passes. Typically scheduled to run daily or weekly to keep user access clean.

GRC Access Control — Risk Analysis (ARA)
Q15–24
Q15 What are the 3 types of risks in GRC? ▼
  • SOD Risk (Segregation of Duties) — Two or more conflicting functions are combined. Example: one person can both create a Purchase Order and approve it. These always involve at least two functions.
  • Critical Action — The T-code itself is high-risk. Examples: SM30 (table change), SCC5 (client deletion), SU25, SE38 (ABAP editor).
  • Critical Permission — A specific authorization object with sensitive values. Examples: S_DEVELOP with debugging + change, S_TABU_DIS, all SU01 and PFCG-related objects.
Q16 What is the difference between Remediation and Mitigation? ▼
Remediation — You remove the risk. Either take away the conflicting T-code from the user/role, or split the access between two people so no single person holds both sides of the conflict.

Mitigation — You accept the risk. The business has reviewed it and decided the access is necessary. You document it in GRC, assign a mitigation control and owner, and set a validity period. Default validity is 1 year.

Mitigation doesn't remove the risk — it means someone is aware of it and monitoring it.
Q17 What is the difference between User Level and Role Level Simulation? ▼
User Level Simulation — Use this when you're about to assign a role to a user. It checks: "If I add this role, will the user get any new SOD risks combined with their existing access?" You're testing at the individual user level.

Role Level Simulation — Use this when you're about to add a T-code inside a role. It checks if the role itself will have a risk after the change.

Both must be done before making any changes — it's a mandatory approval step.
Q18 What is a Risk ID in GRC and how do you create one? ▼
A Risk ID defines a specific business risk in GRC. Go to NWBC → Access Risk Management → Access Risk and fill in:
  • Risk ID & Description
  • Risk Type — SOD, Critical Action, or Critical Permission
  • Risk Level — Low / Medium / High / Critical
  • Status — Active or Inactive
  • Function IDs — The conflicting functions. SOD risks need at least two functions. Each function contains T-codes (Actions) and objects (Permissions).
Save it and the risk is available for future risk analysis.
Q19 What is the difference between Online and Offline Risk Analysis? ▼
Online Risk Analysis — Fetches data live from the backend ECC system. Always reflects the most current state of user access. Most accurate.

Offline Risk Analysis — Runs against data already stored in GRC tables (populated via sync jobs). Faster because it doesn't call the backend, but data could be slightly outdated if sync jobs haven't run recently. Useful for bulk periodic reviews.
💡 Always use Online for individual requests. Use Offline for large-scale reviews where speed matters more than real-time accuracy.
Q20 What is the difference between Full Sync and Incremental Sync? ▼
Full Sync — Copies everything from the backend (ECC) to GRC tables. Slower but gives a complete, accurate picture.

Incremental Sync — Only copies what is new or changed since the last sync. Much faster, but depends on the previous sync having been complete.
💡 Best practice: Run a full sync periodically (e.g., monthly) and use incremental sync for daily updates.
Q21 What background sync jobs run in GRC? ▼
  • Repository Object Sync — Moves users, roles, and profiles from ECC into GRC tables. Required for offline risk analysis and raising access requests.
  • Authorization Sync — Pushes SU24 data (T-code to object mappings) into GRC tables.
  • Role Usage Sync — Tracks which roles are actually being used.
  • Action Usage Sync — Tracks which T-codes are being used and by whom.
  • FF Workflow Sync — Handles Firefighter log reporting and email notifications to controllers.
Q22 How do you update Role Owners in GRC, including mass updates? ▼
Individual update: Go to NWBC → BRM → Search Roles, filter by the current owner who left, get the list of roles, and update each one with the new owner's user ID.

Mass update: Download the GRC template, fill in role names and new owner IDs in bulk, then upload it back. Much faster when dozens of roles need updating at once.
Q23 How does the ARM Access Request process work end-to-end? ▼
1
User raises a ticket requesting access to a T-code
2
Security team searches SUIM to find the right role containing that T-code
3
Create GRC Access Request in NWBC → ARM → Create Request (enter user ID, role, request type)
4
Request goes to Manager for approval
5
Then goes to Role Owner for approval
6
Security stage — we run user-level risk simulation. If risk exists, mitigate with business approval
7
Once all stages approve, access is automatically provisioned in the backend
Q24 How do you import roles from the backend into GRC (BRM)? ▼
Go to NWBC → BRM → Role Import. Fill in role type (Technical or Business Role), source system connector, and role name. Submit. The role now appears in GRC and can be selected in access requests.

For a single role — do it manually. For multiple roles — use the bulk upload template.
💡 Always run Role Import after creating a new role in the backend. Without it, the role won't show up in GRC access request forms.

EAM — Emergency Access Management (Firefighter)
Q25–31
Q25 What is a Firefighter ID and why do we need it? ▼
A Firefighter ID is a special shared account that holds emergency or critical access — the kind that's too risky to give to a regular user ID permanently.

When something urgent happens — like a month-end job fails or emergency config changes are needed — the user's own ID doesn't have the required access. Instead of permanently expanding their access, we let them temporarily use a Firefighter ID for that specific task. Everything done under the FF ID is logged and reviewed by the FF Controller.
💡 FF IDs are Service type users — password doesn't expire, multiple logins are allowed, and they're not permanently tied to any one person.
Q26 User-Based vs Role-Based Firefighter — which is recommended and why? ▼
User-Based FF — User is assigned a specific FF ID (e.g., FF_PROD01). They log into that separate ID. Very clear: any action during the FF session belongs to the FF ID, not their personal access.

Role-Based FF — An FF role is added to the user's own ID. The problem: neither the user nor the security team can tell if an action was done through their normal access or through the FF role.

SAP recommends User-Based because it gives a clean, unambiguous audit trail.
Q27 What are the key EAM configuration parameters? ▼
  • 4000 — Application Type: User-Based or Role-Based Firefighter
  • 4001 — Validity: How long a FF session stays valid (e.g., 24 hours)
  • 4010 — FF Role: The role assigned to FF IDs in the backend (SAP_GRAC_SPM_FFID)
  • 4015 — Centralized vs Decentralized: Whether FF access is managed through GRC (centralized) or directly in the backend (decentralized)
Q28 What is the difference between Centralized and Decentralized Firefighter? ▼
Centralized FF — User logs into GRC first (T-code GRAC_EAM or GRAC_SPM), then launches the FF session in the backend from there. All logging and workflow are managed in GRC.

Decentralized FF — User logs directly into the backend system (ECC) and accesses the FF ID from there. GRC is not in the middle of the session.
Q29 What are the 4 steps to configure a Firefighter ID in EAM? ▼
1
Create the FF ID in the backend system as a Service user with the appropriate access
2
Define Owners and Controllers in GRC via Access Control Owners
3
Map the FF ID to its Owner and Controller in GRC
4
Define Reason Codes so users must provide a valid reason when requesting FF access
Q30 How does the end-to-end Firefighter process work for a user? ▼
1
User raises FF request in NWBC — selects the appropriate FF ID and provides a reason code
2
Request goes to the FF Owner for approval
3
Once approved, the user receives an email notification
4
User logs into the FF session and performs the required emergency activities
5
Session ends — logs are automatically sent to the FF Controller for review
6
Controller reviews the activity and closes the ticket
Q31 What is the role of the FF Controller? ▼
The FF Controller is responsible for reviewing what was done during a Firefighter session. After the FF session ends, the system automatically sends activity logs to the controller via workflow. The controller reviews the actions taken and closes the review ticket. This is the audit trail that compliance and audit teams rely on.

MSMP Workflow Configuration
Q32–34
Q32 What is MSMP and what are the 7 steps to configure it? ▼
MSMP stands for Multi-Stage, Multi-Path. It's the GRC workflow engine that automatically routes access requests to the right approvers based on rules you define. Instead of manually sending approval emails, MSMP decides who approves, in what order, and for which type of request.

The 7 configuration steps:
1
Process Global Settings — Overall workflow behavior
2
Maintain Rules — Routing rules (e.g., "New Access → Path X")
3
Maintain Agents — Who the approvers are (e.g., GRC Manager = user's direct manager)
4
Variables & Templates — Email templates and dynamic variables
5
Maintain Paths — Define the approval stages (Manager → Role Owner → Security)
6
Maintain Route Mapping — Link rules to paths
7
Generate Version — Activate and publish the configuration
💡 Custom conditions (e.g., route differently based on risk level or department) can be added using BRF+ integration.
Q33 Can you have more than one initiator rule per Process ID in MSMP? ▼
No — each Process ID can only have one initiator rule. The initiator rule is what kicks off the workflow. Having multiple initiators for the same process would cause ambiguity about where to start. If you need complex routing, you define that logic inside the path and route mapping steps.
Q34 What are the standard approval stages in GRC ARM workflow? ▼
The standard three-stage approval flow:
  • Manager — Reviews and approves the business need for the access
  • Role Owner — Verifies the role is appropriate for this specific user
  • Security Consultant — Reviews risk analysis, checks for SOD conflicts, and either approves or requests mitigation
Once all stages approve, access is automatically provisioned in the backend.

Troubleshooting & Day-to-Day Operations
Q35–41
Q35 A user says they're getting an authorization error. Walk me through your full approach. ▼
1
Ask user to run SU53 immediately after the error — before clicking anything else
2
If SU53 shows the right error → find the object, check which role has it in SUIM, proceed with fix
3
If SU53 doesn't capture the right error → screen-share with user, run STAUTHTRACE live
4
Enable system-wide trace in STAUTHTRACE. User reproduces the error. Turn off trace and evaluate report.
5
Check Return Codes — RC 4 = values missing, RC 12 = object missing, RC 8 = ask user to re-login
6
Get business approval, then follow standard role assignment or change process
Q36 How do you handle an SLA breach on a ticket? ▼
Every ticket has two SLA clocks — Response SLA (how fast you acknowledge) and Resolution SLA (how fast you close it).

When an SLA is breached:
  • Analyse why it breached — late assignment? Dependency on Basis team for transport? Waiting on business approval?
  • Document the reason clearly in the ticket as a breach justification
  • For access tickets — fast-track approvals, get on a direct call with the user/manager
  • Keep the user informed throughout — proactive communication is key
  • After resolving, share a plan with the team on how to prevent it going forward
Standard SLA: P1 = 2 hrs · P2 = 2 days · P3 = 5 days · P4 = 10 days.
Q37 The GRC system is not working. What do you check first? ▼
Most common cause: RFC connection failure between GRC and the backend ECC system. If the RFC is down, GRC can't communicate with the satellite system — access requests can't be provisioned, risk analysis fails, and sync jobs stop.

First check: verify the RFC connection is active and the system is reachable. RFC configuration is a Basis activity, so raise it to the Basis team immediately.
Q38 What are your regular day-to-day activities? ▼
  • User Administration — Creating users, password resets, locking/unlocking during monthly maintenance windows
  • Role Maintenance — Handling role change requests (adding/removing T-codes), following Dev → QA → Prod transport process
  • Troubleshooting — Resolving authorization issues via SU53, STAUTHTRACE, SUIM
  • GRC Activities — Risk analysis, updating role owners, user/role simulation, importing roles via BRM
  • Audit Reports — Quarterly user access reviews (UAR), inactive users report (90+ days), critical auth checks (e.g., SAP_ALL holders via UST04)
Q39 What key SAP security tables should you know? ▼
  • AGR_DEFINE — Master/Derived role relationship
  • AGR_1252 — Org level values in a role
  • AGR_AGRS — Single roles inside a composite role
  • AGR_USERS — User to role assignments
  • UST04 — User to profile assignment (find SAP_ALL holders)
  • USOBT_C / USOBX_C — Customer SU24 data (T-code to object mapping)
  • TDDAT — Table to authorization group mapping (for S_TABU_DIS)
  • SM20 — Security audit log
Q40 How do you delete an authorization object in a role in PFCG? ▼
In PFCG, authorization objects have 4 possible statuses. Only 2 statuses allow direct deletion. You cannot delete an active object without deactivating it first.

Correct sequence:
  • Deactivate the authorization object in the role first
  • Then delete it
Always regenerate the profile after making changes.
