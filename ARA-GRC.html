AC-01

GRC 10.1 vs GRC 12

Almost always the opening question — interviewers use it to check whether you understand why organisations upgrade, not just that they do.

Q1What's the difference between GRC 10.1 and GRC 12, and why upgrade?+

Frame it as three buckets — interface, performance, integration:

  • Interface: 10.1 runs on the classic SAP GUI / NetWeaver Business Client. 12 runs on Fiori Launchpad — create request, user-level analysis, and simulation are all tile-based, which is simply a better user experience.
  • Performance: 12 leverages the SAP HANA in-memory database, so risk analysis and reporting run faster. 10.1 sits on the older NetWeaver Application Foundation.
  • Integration: 10.1 has limited third-party integration. 12 can integrate with cloud platforms like SuccessFactors and Ariba, which matters as more clients move HR and procurement to the cloud.
How to say itLead with the bucket structure out loud — "I'd compare it across UI, performance, and integration" — then fill each one in a sentence. It signals structured thinking before you've said a single technical detail.
AC-02

Components & modules

Interviewers use this to confirm you know GRC's map before drilling into any one room of the house.

Q2What are the different components of GRC?+

Three main pillars come up most often in interviews:

  • Access Control (AC) — access risk and provisioning.
  • Process Control (PC) — internal controls and compliance monitoring.
  • Global Trade Services (GTS) — trade compliance.

Environment, Health & Safety also exists in the GRC suite, but it rarely comes up unless the role specifically touches it — mention it briefly if asked "is that all," don't lead with it.

Q3What are the four modules under GRC Access Control?+
  • ARM — Access Request Management
  • ARA — Access Risk Analysis (also written ARRA)
  • EAM — Emergency Access Management
  • BRM — Business Role Management
How to say itSay the acronym and expand it in the same breath: "ARM, which is Access Request Management…". It reads as fluency rather than a memorised list.
AC-03

Rule set fundamentals

This is the conceptual core of ARA — get the definitions precise, because every later question builds on them.

Q4Explain the rule set. What does it contain?+

A rule set contains access risks. An access risk is normally two or more conflicting functions — that's a classic segregation-of-duties (SoD) risk. A single function on its own isn't an SoD risk, but it can still be flagged as a critical action risk if that one function alone is sensitive enough.

Each function, in turn, is made up of actions (t-codes) and permissions (authorization objects and values). So the hierarchy is:

  • Rule set → contains access risks
  • Access risk → one or more conflicting functions
  • Function → actions (t-codes) + permissions (auth objects/values)
How to say itDraw the hierarchy verbally, top-down: "rule set contains risks, risks are built from functions, functions are built from t-codes and authorizations." That one sentence answers three follow-up questions before they're asked.
Q5What is a Rule ID / rule?+

A Rule ID is a unique identifier assigned to a specific combination of conflicting actions or permissions under one risk — essentially the fingerprint of that particular conflict combination, visible in the detailed risk analysis report.

AC-04

Risk analysis report — options & columns

Interviewers probe whether you've actually run risk analysis, not just read about it — know the screen, not just the theory.

Q6When you run user-level risk analysis, what options and columns should you know?+

Include mitigated risks — a checkbox that, when ticked, shows risks even where a mitigation control is already assigned, so you see the full exposure rather than only unmitigated risk.

Summary-level columns:

  • User ID
  • Rule ID (the conflict combination, e.g. 0080)
  • Risk level
  • Action last executed / execution account
  • Mitigation control ID, if the risk is mitigated

Detail-level adds: role name, the specific authorization object and its values, and the system. The same summary/detail structure applies to role-level risk analysis too — just swap "user" for "role."

How to say itYou don't need to recite every column from memory — say "summary gives you the who and what risk, detail drills into which role and which authorization object caused it," then name two or three columns as examples. Depth of understanding beats a memorised list.
AC-05

Mass mitigation

A practical, hands-on question — good for showing you've actually run remediation, not just analysis.

Q7How do you perform mass mitigation of risks?+

Through transaction SA38, using the program to upload or download mitigation control assignments — commonly referred to as the "upload/download mitigation assignments" program. You add the mitigation control ID and the list of users who need it, then upload the file so the assignments apply in bulk rather than one user at a time.

How to say itMention the download-then-edit-then-upload cycle — it shows you understand this is templated, auditable, and repeatable, not a manual workaround.
AC-06

Offline vs online risk analysis

Q8What is offline risk analysis vs online risk analysis?+
  • Offline (batch) risk analysis: runs as a background job against previously collected data rather than live data — you see it labelled "batch risk analysis" in the Access Risk Analysis screen. Useful for large-scale, scheduled reporting.
  • Online risk analysis: run live, typically at the user level, and reflects current data at the moment you execute it.
AC-07

False positive & false negative risk

The single most-asked scenario question in ARA interviews — expect a follow-up asking you to walk through a real example.

Q9What is a false positive vs a false negative risk? Give an example.+

False positive: the report flags a risk that isn't actually a risk in business terms. Classic example — a user has "Approve PO" in one role, restricted to company code 1000, and "Create PO" in a second role, restricted to company code 2000. Standard SoD analysis looks only at the t-codes, sees both actions on the same user, and flags a conflict — even though in practice the user can never approve a PO they created, because the company codes never overlap.

False negative: the opposite problem — a genuine risk exists, but it doesn't appear in the report at all, usually because the rule set is incomplete or missing values for that combination.

Why it mattersFalse positives create noise that erodes trust in the SoD report; false negatives create blind spots. Fixing false positives is what organizational rules and supplementary rules exist for — see AC-08.
How to say itWalk through the PO example out loud with the two company codes — a concrete scenario lands far better than reciting the definition, and it's almost guaranteed to be the follow-up question anyway.
AC-08

Organizational rules & supplementary rules

The remediation half of AC-07 — knowing the fix is what separates a theoretical answer from a practitioner's answer.

Q10How do organizational rules fix false positives? Walk through the config.+

Organizational rules teach the risk engine to respect an organizational value — like company code — instead of matching purely on t-code.

  • Create the org rule (e.g. Z_OR_1), optionally linking a parent org rule.
  • Select the access risk ID the rule applies to and add a description.
  • Maintain the organizational level (e.g. company code) with a from–to value range and the condition.
  • Specify the system the rule applies to, then save.
  • In risk analysis itself, tick "Consider Org Rules" — without this checked, the org rule is ignored even though it's configured.

Once set up correctly, a user with Approve PO for company code 1000 and Create PO for company code 2000 no longer shows as a risk.

Q11What are supplementary rules used for?+

Supplementary rules exclude specific users or values from analysis entirely — for example, a release-strategy t-code combination that should be flagged for everyone except a defined group of approvers. You maintain a table listing those exceptions, specify the system, choose the relevant field (typically username), add the field, and execute — the risk analysis then runs for everyone except the users in that table.

How to say itContrast the two in one line: "org rules filter by organizational value like company code; supplementary rules filter by a defined exception list of users." That contrast is usually exactly what the interviewer is listening for.

AC-10

Key tables for access risk analysis

You're not expected to know every field — knowing the naming convention and two or three specific tables shows real hands-on exposure.

Q13What tables are commonly used in access risk analysis?+

Most access-risk-analysis tables start with GRAC or GRFN — that naming convention alone is worth stating out loud, it shows pattern recognition rather than rote memorisation.

Two tables worth naming specifically:

  • HRT5320 — used to see mitigation control ownership: who the assigned owner, controller, monitor, and approver are for a given mitigation.
  • HRP5354 (and related HRP53xx object-id tables) — hold the object IDs that link into HRT5320 to resolve the monitor and approver for a specific mitigation object.

You can also query directly for SoD risk owners — who is set as the owner for a given access risk ID — which is configured separately and often left blank in smaller implementations.

AC-11

Risk check at request submission

Q14Can risk analysis run automatically when an access request is submitted?+

Yes. Rather than relying on someone manually running risk analysis afterward, you can enable automatic SoD risk analysis at the point of request submission via parameter 1071 in SPRO → GRC → Access Control → Maintain Configuration Settings — labelled "Enable risk analysis upon request submission" / "Enable SoD violations." This surfaces conflicts to the approver at request time, before access is even granted.

How to say itTie it back to the business value in one sentence: "it shifts SoD checking left, so conflicts are caught before provisioning instead of discovered in an audit months later."
